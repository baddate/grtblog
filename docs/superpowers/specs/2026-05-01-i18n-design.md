# i18n Internationalization Design

## Overview

Remove all hardcoded strings (~5,600 occurrences) from the goblog monorepo and add i18n
support for Chinese (zh) and English (en), with an extensible framework for future
languages.

### Key Decisions

| Decision | Choice |
|----------|--------|
| Languages | zh + en, framework extensible for future |
| Language detection | URL path (`/zh/`, `/en/`) primary; Accept-Language for first visit; Cookie for persistence |
| Manual switcher | Footer language toggle on all pages |
| Admin i18n | Yes, full platform coverage |
| Translation key strategy | Hierarchical namespaced keys (`common.save`, `server.error.article.not_found`) |
| Migration strategy | Phased: Web first (~600 strings), then Server (~2,200), then Admin (~2,800) |

### Current State

| Component | Files | Chinese occurrences | English errors | i18n infra |
|-----------|-------|--------------------|----------------|-----------|
| web (SvelteKit) | ~116 | ~602 | few | none |
| server (Go) | 174 | ~2,189 | ~379 | none |
| admin (Vue 3) | ~182 | ~2,779 | few | none |
| telemetry-server (Go) | 1 | 13 | ~50 | none |

telemetry-server is intentionally excluded: it is an internal operational service (13
Chinese strings only in a literary disguise page, English-only API). No i18n
libraries, no locale files, no translation-loading code exists anywhere in the project.

## Architecture

### Translation File Layout

```
goblog/
├── locales/                          # Repo-level shared common UI strings
│   ├── zh/common.json
│   └── en/common.json
│
├── web/src/locales/                  # Web public-facing strings
│   ├── zh/web.json
│   └── en/web.json
│
├── admin/src/locales/                # Admin dashboard strings
│   ├── zh/admin.json
│   └── en/admin.json
│
└── server/locales/                   # Server API response strings
    ├── zh/server.json
    └── en/server.json
```

- `common.*` — shared by all three components (~50 keys: save, cancel, confirm, delete, etc.)
- `web.*` — SvelteKit public-facing only (SEO, brand copy, search, date formatting)
- `admin.*` — Vue admin only (menus, form labels, table headers, settings panels)
- `server.*` — Go API responses only (domain errors, BizError messages, validation)

### Translation Key Convention

Format: `{domain}.{module}.{component}.{field}`

Examples:
```
common.save                    → 保存 / Save
common.cancel                  → 取消 / Cancel
web.article.reading_time       → {n} 分钟阅读 / {n} min read
web.search.placeholder         → 探索知识与灵感... / Explore knowledge...
admin.article.edit.title       → 编辑文章 / Edit Article
admin.settings.site.title      → 站点信息 / Site Info
server.error.article.not_found → 文章不存在 / Article not found
server.error.auth.not_login    → 未登录或登录已过期 / Login expired
```

### JSON Format

Flat JSON with support for interpolation and pluralization:

```json
{
  "web.article.reading_time": "{n} min read",
  "web.comment.count": "{count, plural, one {# comment} other {# comments}}"
}
```

## Language Routing & Detection

### Flow

```
User first visit → URL has /zh/ or /en/ prefix?
    ├── YES → Use that language, set Cookie
    └── NO
        ├── Has Cookie preference? → 302 redirect to /{lang}/xxx
        └── NO
            ├── Read Accept-Language header
            ├── zh preferred → 302 to /zh/xxx
            └── otherwise → 302 to /en/xxx
                            Set-Cookie: lang=en
```

### URL Structure

```
/en/                          → Home (English)
/en/articles/                 → Article list
/en/articles/some-post/       → Article detail
/zh/                          → Home (Chinese)
/zh/articles/                 → Article list
/zh/articles/some-post/       → Article detail
/admin/                       → Admin (independent, no lang prefix)
/api/                         → API (independent, no lang prefix)
```

### Per-Component Language Resolution

| Component | Detection mechanism |
|-----------|-------------------|
| Web (SvelteKit) | URL prefix `[lang]` param, parsed in `hooks.server.ts` |
| Admin (Vue) | Pinia `useLocaleStore` persisted to localStorage, default from `navigator.language` |
| Server (Go) | `Accept-Language` header or `?lang=` query param, via Fiber middleware |

### Footer Language Switcher

Placed at the bottom of every public page. Switches the `[lang]` prefix in the current
URL path and updates the Cookie.

## Per-Component Implementation

### Phase 1: Web (SvelteKit)

No external i18n library — lightweight custom solution to avoid bundle bloat in SSR/SSG.

```
src/hooks.server.ts
  └── Parse [lang] from URL
  └── Load locale JSON (common + web), merge
  └── Inject event.locals.lang, event.locals.t()

src/lib/i18n/
  ├── index.ts           # Public API: t() function, types
  ├── server.ts          # Server-side translation loader
  ├── client.ts          # Client-side (sync from server data)
  └── types.ts           # TranslationKey union type

src/routes/[lang]/       # New route structure
  ├── +layout.server.ts  # Load translations from params.lang
  └── (all existing page routes moved here)
```

Usage in Svelte components:

```svelte
<script>
  let { data } = $props();
</script>
<button>{data.t('common.save')}</button>
<p>{data.t('web.article.reading_time', { n: 5 })}</p>
```

**SSG consideration:** `adapter-static` prerenders each language independently. The
existing prerender configuration (trailingSlash: 'always') is preserved.

### Phase 2: Server (Go)

Library: `nicksnyder/go-i18n/v2` (most mature Go i18n library).

```
server/locales/
  ├── zh/server.json
  └── en/server.json

server/internal/infra/i18n/
  ├── i18n.go          # Bundle init, load locales + common
  ├── middleware.go     # Fiber middleware: Accept-Language parsing
  ├── keys.go           # Generated key constants (compiler-safe)
  └── i18n_test.go
```

Usage in handlers:

```go
localizer := i18n.GetLocalizer(c)
msg := i18n.MustLocalize(localizer, "server.error.article.not_found")
return response.NewBizErrorWithMsg(c, response.NotFound, msg)
```

**Domain error refactoring:** Sentinel errors change from Chinese messages to
machine-readable codes. The human-readable message is resolved at the HTTP layer via
the i18n bundle.

```go
// Before
var ErrArticleNotFound = errors.New("文章不存在")

// After — sentinel is a code, message comes from i18n
var ErrArticleNotFound = errors.New("ARTICLE_NOT_FOUND")
```

The global Fiber ErrorHandler in `server.go` maps these sentinel errors to their
`BizError` code, and the i18n middleware resolves the message.

### Phase 3: Admin (Vue 3)

Library: `vue-i18n` v10 (Vue ecosystem standard, native Naive UI integration).

```
admin/src/locales/
  ├── zh/admin.json
  └── en/admin.json

admin/src/plugins/i18n.ts
admin/src/stores/locale.ts     # Pinia store: current locale + localStorage persistence
```

Naive UI integration:

```ts
import { createI18n } from 'vue-i18n';

const i18n = createI18n({
  locale: savedLocale || navigator.language,
  messages: {
    zh: { ...commonZh, ...adminZh },
    en: { ...commonEn, ...adminEn },
  },
});
```

Usage in Vue components:

```vue
<template>
  <NButton>{{ $t('common.save') }}</NButton>
  <h2>{{ $t('admin.article.edit.title') }}</h2>
</template>
```

## Build & Merge Pipeline

Each component merges common + component-specific locales at build time.

**Web/Admin:** `scripts/merge-locales.mjs` reads `locales/{lang}/common.json`, merges
with component locales, writes to a gitignored generated directory. TypeScript types for
`TranslationKey` are generated from the merged JSON keys.

**Server:** Go `//go:embed` directive loads both `locales/` (repo root) and
`server/locales/` at init time into the i18n bundle.

### CI Checks

| Check | Method |
|-------|--------|
| Key symmetry (zh ↔ en) | Script compares key sets; fails if any key exists in only one language |
| Placeholder parity | Regex check that `{n}`, `{count}` placeholders match across languages |
| Empty value detection | Any value `""` or value equal to its key → failure |
| Valid JSON | `json_pp` or `jq` syntax check |
| Key reference validity | Generated TypeScript types (web/admin) and Go constants (server) enforce at compile time |

## ISR Interaction

Content changes trigger ISR for all supported languages:

```
Content change
  → Go ISR scheduler computes affected paths
  → Before: ["/articles/some-post/"]
  → After:  ["/zh/articles/some-post/", "/en/articles/some-post/"]
  → Requests SvelteKit renderer with lang parameter
  → Writes: static/zh/articles/some-post/index.html
            static/en/articles/some-post/index.html
  → WebSocket broadcast includes lang, clients refresh only matching language
```

## Error Handling & Fallbacks

### Missing Translation

| Context | Behavior |
|---------|----------|
| Development | `console.warn` + display `⚠ key.name` prominently |
| Production | Fallback chain: target lang → en → key itself |
| CI | Build fails (key asymmetry or empty value) |

### Edge Cases

| Scenario | Fallback |
|----------|----------|
| Unsupported lang in URL (`/fr/`) | 302 redirect to en |
| Unparseable Accept-Language | Default to en |
| API request without Accept-Language | Default to en |
| Naive UI built-in strings | Naive UI's own locale packs handle these |

## Migration Plan

### Phase 1: Web (~600 strings)

1. Create `locales/{zh,en}/common.json` (~50 shared keys)
2. Create `web/src/locales/{zh,en}/web.json`
3. Add `scripts/merge-locales.mjs` + type generation
4. Restructure routes under `src/routes/[lang]/`
5. Implement `hooks.server.ts` language detection
6. Implement `src/lib/i18n/` with `t()` function
7. Migrate strings by priority: SEO metadata → date formatting → layout/navigation → search → page content
8. Add footer language switcher component

### Phase 2: Server (~2,200 strings)

1. Add `nicksnyder/go-i18n/v2` to go.mod
2. Create `server/locales/{zh,en}/server.json`
3. Implement `internal/infra/i18n/` bundle + middleware
4. Add key constant generation
5. Refactor domain error sentinel values to machine-readable codes
6. Migrate BizError messages
7. Migrate handler-level `NewBizErrorWithMsg` calls
8. Update ISR scheduler for dual-language paths
9. Update global ErrorHandler for i18n-aware error mapping

### Phase 3: Admin (~2,800 strings)

1. Add `vue-i18n` v10 to admin dependencies
2. Create `admin/src/locales/{zh,en}/admin.json`
3. Implement `plugins/i18n.ts` + `stores/locale.ts`
4. Integrate Naive UI locale packs
5. Migrate route labels in `router/record.ts`
6. Migrate layout: sidebar, header, tabs, preferences
7. Migrate view-level strings by domain (settings → articles → moments → comments → ...)

## Testing Strategy

| Layer | Test type | What it verifies |
|-------|-----------|-----------------|
| CI | `make i18n-check` | Key symmetry, placeholder parity, empty values, valid JSON |
| Web | Vitest unit tests | `t()` resolves keys correctly, fallback chain works |
| Web | E2E (Playwright) | `/zh/` and `/en/` pages render correct language |
| Server | Go test | `MustLocalize` returns correct messages, Accept-Language parsing |
| Admin | Vitest unit tests | vue-i18n pipe resolves keys, locale store persistence |
| All | TypeScript/Go compilation | Key constants/types guarantee referenced keys exist |
